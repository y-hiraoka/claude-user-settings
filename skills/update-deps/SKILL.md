# 依存パッケージ更新スキル

このスキルは、リポジトリが依存しているパッケージを安全にアップデートするためのワークフローを定義する。
闇雲にバージョンを上げるのではなく、変更内容の調査・テストによる動作保証・段階的なコミットを通じて、安心してアップデートできることを目指す。

## ワークフロー概要

1. 更新対象の特定
2. 各パッケージの Changelog / リリースノート調査
3. リポジトリ内の利用箇所の特定
4. テストの充足度を判断し、不足していればテストを書く
5. テストコードをコミット（動作保証の証拠）
6. パッケージを更新し、必要に応じてコードを修正
7. テストが通ることを確認してコミット

### patch バージョンのショートカット

patch バージョン（例: `1.2.3` → `1.2.4`）の更新は、semver の規約上バグ修正のみで後方互換性が保たれるため、Step 4（テスト充足度の判断）と Step 5（テストコードのコミット）をスキップしてよい。Step 1 → 2 → 3 → 6 → 7 の流れで進め、Step 7 では既存テストの実行のみで動作確認とする。

ただし、すべてのパッケージが semver を厳密に守っているとは限らない。Step 2 の Changelog 調査で patch にもかかわらず破壊的変更や挙動の変更が含まれていることが判明した場合は、ショートカットせず通常フロー（Step 4-5 含む）に戻すこと。

## Step 1: 更新対象の特定

### outdated パッケージの確認

```bash
pnpm outdated -r
```

- monorepo の場合は `-r`（recursive）オプションを付ける。単一パッケージのリポジトリでは `-r` は不要。
- 出力は patch バージョンの差分が小さいものから表示される。**上から3つ**のパッケージを今回の更新対象の候補とする。
- 対象は dependencies / devDependencies の両方を含む。

### 関連パッケージの確認と更新対象の絞り込み

上から3つの候補を選んだら、各パッケージに**密接に関連する他のパッケージ**がないかを確認する。

たとえば `zod` は `@hono/zod-openapi` や `@hono/zod-validator` と密接に関連しており、`zod` だけをバージョンアップすると関連パッケージの使用箇所が壊れる可能性がある。このような場合、関連パッケージも同時に更新する必要がある。

関連パッケージを含めた結果、更新対象が膨らみすぎないように以下のルールで絞り込む：

- 候補パッケージに関連パッケージがある場合、「候補パッケージ + 関連パッケージ」を1つのグループとして扱う。
- グループ化した結果、**更新するパッケージの総数が最小になるように**候補を選ぶ。
- 具体的には、上から順に候補を選んでいき、3つ目の候補をグループ化すると総数が大きく膨らむ場合は、3つ目を諦めて上2つ（+それぞれの関連パッケージ）に専念する。
- 逆に1つ目の候補が大きなグループになる場合は、その1グループだけを今回の更新対象とする。

**例:**

- 上から `A`（単独）、`B`（単独）、`zod`（+ `@hono/zod-openapi`, `@hono/zod-validator`）の場合 → `A` + `B` の2つに専念。`zod` グループを含めると総数が5になるため。
- 上から `zod`（+ `@hono/zod-openapi`, `@hono/zod-validator`）、`A`、`B` の場合 → `zod` グループ（3パッケージ）を更新。`A`, `B` はカウントしない。

### pnpm catalog の確認

monorepo の場合、`pnpm-workspace.yaml` に `catalog:` セクションがあるかを確認する。

- **catalog で管理されているパッケージ**: 複数ワークスペースで使われていても catalog のバージョンを1箇所更新するだけで全ワークスペースに反映される。まとめて更新する。
- **catalog を使わず各ワークスペースが個別にインストールしているパッケージ**: ワークスペースごとに個別に処理する。

## Step 2: Changelog / リリースノートの調査

更新対象の各パッケージについて、現在のバージョンからターゲットバージョンまでの変更内容を調査する。

### 調査方法

- WebSearch や WebFetch を使って、パッケージの GitHub リリースページや CHANGELOG.md を確認する。
- npm のパッケージページ (`https://www.npmjs.com/package/<name>`) にもリリース情報がある場合がある。
- 調査結果をユーザーに簡潔に報告する。特に以下の点に注目：
  - **Breaking changes**: API の変更、削除された機能、挙動の変更
  - **Deprecations**: 将来のバージョンで削除予定の機能
  - **新機能**: 活用できそうな新しい API やオプション
  - **バグ修正**: リポジトリに影響しそうな修正

### devDependencies の場合

devDependencies（ビルドツール、リンター、テストフレームワークなど）の場合は、ランタイムのコードに直接影響しないため、テストコードの追加よりも以下に注力する：

- 設定ファイルの変更が必要か（例: ESLint の設定形式変更）
- `package.json` の `scripts` に影響があるか
- ビルドやテストのコマンドに変更が必要か

## Step 3: リポジトリ内の利用箇所の特定

Grep や Glob を使って、更新対象パッケージがリポジトリ内でどのように使われているかを調べる。

- `import` / `require` 文の検索
- 設定ファイルでの参照
- 変更される API を使用している箇所の特定

影響範囲を把握することで、次のステップでテストの充足度を正しく判断できる。

## Step 4: テストの充足度を判断

Step 2 で把握した変更内容と Step 3 で特定した利用箇所を突き合わせて、既存のテストが十分かどうかを判断する。

### テストが十分と判断する基準

- 変更される API の呼び出しがテストでカバーされている
- Breaking changes に該当するコードパスがテストされている
- パッケージ更新後にテストが壊れれば問題に気づける状態

### テストが不足している場合

- 現在の動作を保証するテストを**先に**書く。パッケージ更新の前にテストを書くことが重要。更新前のテストがグリーンであることが「現在の動作」の証拠になる。
- テストは最小限でよい。パッケージの変更に関連する部分だけをカバーすれば十分。

### テストコマンドの確認

リポジトリで使用されているテストコマンドを確認する。`package.json` の `scripts` セクションや、README、CLAUDE.md などを参照する。単体テストだけでなく、E2E テストが存在する場合はそちらも確認対象に含める。

### devDependencies の場合

devDependencies はランタイムコードに影響しないことが多いため、テストコードの追加は不要な場合がある。その場合はこのステップをスキップして Step 6 に進む。

## Step 5: テストコードのコミット

テストを新たに書いた場合は、パッケージ更新の前にコミットする。

- コミットメッセージは、テストを追加した意図（依存更新に備えた動作保証）が伝わるようにする。
- このコミットが「更新前の動作が正しかった」ことのエビデンスになる。

## Step 6: パッケージの更新とコード修正

### パッケージの更新

```bash
# 特定のパッケージを更新
pnpm update <package-name>

# catalog で管理されている場合は pnpm-workspace.yaml のバージョンを直接編集して pnpm install
```

### コード修正

Changelog 調査で判明した Breaking changes や Deprecations に基づいて、必要なコード修正を行う。

- API の変更に合わせた呼び出し箇所の修正
- 非推奨機能から新しい API への移行
- 設定ファイルのフォーマット変更への対応

## Step 7: テスト実行とコミット

テストを実行して、更新後も正しく動作することを確認する。

- Step 4 で確認したテストコマンドを使ってテストを実行する。
- テストが通ったら、パッケージ更新とコード修正をまとめてコミットする。
- テストが失敗した場合は原因を調査し、修正してからコミットする。

## 注意事項

- 一度に大量のパッケージを更新しない。問題が起きたときに原因を特定しにくくなる。1パッケージずつ、または密接に関連するパッケージをまとめて更新する。
- lock ファイル（`pnpm-lock.yaml`）の変更は必ずコミットに含める。
- パッケージ更新で想定外の大きな変更が必要になった場合は、ユーザーに相談してから進める。
