## 概要

このスキルは、Claude と Codex CLI（`codex exec`）の間でマルチラウンドの議論を行うためのもの。設計書の作成・改善、アーキテクチャ判断、技術的な意思決定などで、2つの AI の視点を突き合わせてより良い結論を導く。

## ワークフロー

### 1. 議論の準備

ユーザーの依頼から以下を整理する:

- **議論の目的**: 何を決めたいのか、何を改善したいのか
- **対象**: 設計書ファイルがあればそのパス、口頭ベースならその内容
- **ラウンド数**: ユーザーが指定していれば従う。指定がなければ「双方が合意に至るまで」続ける（ただし最大 10 ラウンドを目安に収束させる）

### 2. 初回の草案作成

議論対象がまだ草案として存在しない場合は、まず Claude が草案を作成する。既に設計書やドキュメントがある場合はそれを出発点とする。

### 3. Codex との議論ループ

各ラウンドで以下を行う:

#### Codex への依頼

`codex exec` を使って Codex に意見を求める。Codex はステートレスなので、毎回以下を含める:

- 議論の目的と背景
- 現在の草案または論点（ファイルがある場合はファイルパスを伝えて読んでもらう）
- これまでの議論で出た論点と、それぞれの結論
- 今回 Codex に検討してほしい具体的な問い

```bash
codex exec "
あなたは設計レビュアーです。以下の設計について意見をください。

## 目的
（議論の目的）

## 現在の草案
（ファイルパス、または内容の要約）

## これまでの議論経緯
（前のラウンドで合意した点、未解決の論点）

## 今回の相談
（具体的に意見がほしいポイント）
"
```

#### Claude による吟味

Codex の回答を受け取ったら、Claude は以下の姿勢で検討する:

- **妥当な指摘は取り入れる** — Claude が見落としていた観点を Codex が指摘した場合や、良い改善案が出た場合
- **見当違いな指摘は退ける** — Claude のほうがコンテキストを多く持っている場合がある。ユーザーとの会話履歴、プロジェクトの背景、設計上の制約など、Codex に渡していない情報に基づいて判断する
- **建設的に反論する** — 単に「それは違う」ではなく、なぜ違うのか根拠を示し、次のラウンドで Codex に再考を促す

#### 収束判断

以下のいずれかで議論を終了する:

- ユーザーが指定したラウンド数に達した
- 新しい論点が出なくなり、双方の意見が収束した
- 残りの差異が些末で、これ以上議論しても改善が見込めない

### 4. 最終結果の提示

議論が終わったら、ユーザーに最終結果だけを提示する。途中経過は見せない。

提示する内容:

- ブラッシュアップされた設計書やドキュメント（ファイルがある場合は更新済みの内容）
- 口頭ベースの相談の場合は、議論を経て到達した結論のまとめ

## 注意事項

- Codex への毎回のプロンプトには議論の全コンテキストを含める。Codex は前のラウンドの内容を覚えていない
- ファイルの内容をシェルの文字列展開（`$(cat ...)`）で埋め込む必要はない。Codex 自身がファイルを読めるので、パスを伝えるだけでよい
- 議論中に設計書ファイルを更新する場合は、Codex に読んでもらう前に書き込みを完了させる
